# 从“能生成”到“可发布”：给 LLM Wiki 加上一条可靠性链路

LLM 应用最容易被忽略的边界，是模型说“我会返回 JSON”和程序真正拿到可用 JSON 之间的距离。格式围栏、字段缺失、限流和瞬时网络错误都不罕见；如果这些情况散落在业务函数里处理，系统会逐渐变得既难测又难解释。

这次改造把边界收敛成了一条短链路：请求先经过有上限的重试，再解析 JSON 并校验顶层结构；只有通过后，数据才进入 Markdown 渲染和已有的笔记 schema 校验。这样，传输故障、结构故障和领域故障各有明确责任，也各有不同的恢复策略。

Prompt 也被当成代码发布。每类 prompt 都在集中 registry 中拥有 semantic version。修改输出契约时，版本与 eval case 一起变化，评审者就能判断结果漂移来自模型、prompt 还是解析器。

eval 的第一版刻意保持离线：五个固定样例覆盖合法、围栏包裹、字段缺失、类型错误和畸形 JSON。它不能替代语义质量评测，却非常适合作为每次提交都运行的发布门槛。GitHub Actions 在 Windows 与 Node 24 上执行完整单元测试和 eval，不需要任何 API key。

可靠性不是一次“让模型更听话”的 prompt 调整，而是把不确定性逐层缩小：能重试的重试，能确定性校验的交给代码，涉及内容质量的才留给模型修复。最终得到的不是绝不失败的系统，而是失败方式清晰、成本有界、回归可见的系统。
